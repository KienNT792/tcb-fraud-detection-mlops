## 🎯 Project Overview

This project implements a **production-grade MLOps pipeline** for detecting fraudulent credit card transactions. The system covers the full ML lifecycle:

| Stage | Description |
|---|---|
| **Data Ingestion** | Load and validate raw transaction data |
| **Preprocessing** | Clean, engineer features, time-based split |
| **Training** | XGBoost with class imbalance handling + MLflow tracking |
| **Evaluation** | Threshold optimization, SHAP explainability, fairness analysis |
| **Serving** | FastAPI REST API with single & batch prediction |
| **Monitoring** | Data drift detection via Evidently AI, Prometheus + Grafana |
| **Orchestration** | Apache Airflow DAGs for pipeline automation |

### Dataset

| Metric | Value |
|---|---|
| Total transactions | 100,000 |
| Fraud transactions | 2,844 |
| Fraud rate | 2.84% |
| Features | 35 (after engineering) |
| Customer segments | PRIVATE, PRIORITY, INSPIRE, MASS |

> ⚠️ **Class imbalance**: The dataset has an extreme ~97:3 ratio. Accuracy is **never** used as a metric. We optimize for **F1-Score**, **PR-AUC**, **Precision**, and **Recall**.

## 🧪 ML Pipeline

The pipeline is executed sequentially. Each stage produces artifacts consumed by the next.

### 1. Preprocessing (`preprocess.py`)

```bash
cd ml_pipeline && python src/preprocess.py
```

| Step | Description |
|---|---|
| `load_dataset` | Load raw CSV, cast dtypes |
| `validate_schema` | Assert required columns, types, non-null constraints |
| `handle_missing_values` | Fill `os` → "UNKNOWN", `is_3d_secure` → "N", numerics → 0 |
| `clean_data` | Remove duplicates, encode Y/N fields, encode transaction_status |
| `split_dataset` | **Time-based** 80/20 split (prevents temporal leakage) |
| `fit_feature_generators` | Compute customer aggregates **on train only** |
| `transform_features` | Apply log-transform, segment encoding, merge customer stats |
| `save_processed_data` | Write `train.parquet` + `test.parquet` |
| `save_feature_state` | Persist fitted artifacts for inference reproducibility |
| `save_feature_metadata` | Write `features.json` (canonical feature list) |

> 🔒 **Data leakage prevention**: Customer-level aggregates (`customer_tx_count`, `customer_avg_amount`) are computed exclusively from the training set, then applied to the test set via a fit/transform pattern.

### 2. Training (`train.py`)

```bash
cd ml_pipeline && python src/train.py
```

| Step | Description |
|---|---|
| `load_data` | Load processed Parquet + `features.json` |
| `prepare_features` | Extract X/y matrices using only defined features |
| `filter_numeric_features` | Drop non-numeric columns for XGBoost compatibility |
| `compute_class_weight` | `scale_pos_weight = n_negative / n_positive` |
| `train_model` | XGBoost with `aucpr` eval metric + early stopping |
| `evaluate_model` | Compute F1, PR-AUC, ROC-AUC, confusion matrix |
| `save_artifacts` | Persist model (joblib), metrics (JSON), importance (CSV) |

All hyperparameters and metrics are automatically logged to **MLflow**.

### 3. Evaluation (`evaluate.py`)

```bash
cd ml_pipeline && python src/evaluate.py
```

| Step | Description |
|---|---|
| `evaluate_threshold` | Sweep thresholds, find optimal for Recall ≥ 95% |
| `evaluate_segments` | Per-segment Precision/Recall/F1 (fairness check) |
| `explain_shap` | SHAP beeswarm (global) + waterfall (local) plots |
| `compare_baseline` | Regression gate: current vs baseline metrics |
| `save_evaluation_report` | Export `evaluation.json` + `segment_report.csv` |

### 4. Inference (`inference.py`)

```python
from ml_pipeline.src.inference import FraudDetector

detector = FraudDetector("models", "data/processed")
result = detector.predict_single({
    "transaction_id": "TX_001",
    "timestamp": "2026-03-14 10:23:00",
    "customer_id": "CUST_123",
    "amount": 5_000_000,
    "customer_tier": "PRIORITY",
    # ... remaining fields
})
# → {"fraud_score": 0.023, "is_fraud_pred": False, "risk_level": "LOW"}
```

## 🚀 Quick Start

### Prerequisites

- Python 3.10.9
- pip (or virtualenv)

### Installation

```bash
# Clone repository
git clone https://github.com/KienNT792/tcb-fraud-detection-mlops.git
cd tcb-fraud-detection-mlops

# Create virtual environment
python -m venv ml_pipeline/venv
ml_pipeline\venv\Scripts\activate    # Windows
# source ml_pipeline/venv/bin/activate  # Linux/Mac

# Install dependencies
pip install pandas numpy scikit-learn xgboost pyarrow joblib mlflow shap matplotlib evidently
pip install pytest pytest-cov fastapi uvicorn httpx
```

### Run Full Pipeline

```bash
cd ml_pipeline

# Step 1 — Preprocess
python src/preprocess.py

# Step 2 — Train
python src/train.py

# Step 3 — Evaluate (SHAP + Fairness)
python src/evaluate.py

# Step 4 — Inference smoke test
python src/inference.py
```

### Launch API

```bash
cd serving_api
uvicorn app.main:app --reload --port 8000
```

## 🤖 Responsible AI

This project adheres to **Responsible AI** principles as mandated by the grading criteria:

### Explainability (SHAP)

- **Global**: SHAP beeswarm plot reveals which features most influence fraud predictions across all transactions.
- **Local**: SHAP waterfall plot explains *why* a specific high-risk transaction was flagged.
- Artifacts: `models/evaluation/shap_summary.png`, `shap_waterfall.png`

### Fairness

- Model performance is evaluated **per customer segment** (PRIVATE, PRIORITY, INSPIRE, MASS).
- Metrics tracked: Precision, Recall, F1, PR-AUC per segment.
- Artifact: `models/evaluation/segment_report.csv`

### Regression Gate

- Evaluation includes a **baseline comparison** to prevent silent model degradation.
- Tolerance: PR-AUC ≤ 2%, F1 ≤ 3%, Recall ≤ 2% regression.

## 🚢 CI/CD Deployment

The deployment workflow proposed for `dev/ver2` packages the platform with
Docker and deploys it to a Google Cloud VPS through Jenkins.

- `Jenkinsfile` defines the CI/CD stages: checkout, lint, tests, Docker build, SSH deploy, and health checks.
- `docker-compose.yml` brings up the runtime stack: FastAPI, MLflow, MinIO, Airflow, Prometheus, Grafana, plus host/container exporters.
- `dags/fraud_pipeline.py` schedules the nightly `preprocess -> train -> evaluate` retraining pipeline.
- `serving_api/app/main.py` now exposes `/metrics` so Prometheus can scrape API latency and request metrics.

See [docs/cicd_proposal.md](docs/cicd_proposal.md) for the full deployment flow and environment setup.
